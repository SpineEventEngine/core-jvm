Package io.spine.server.aggregate

Class AggregateStorage

  • All Implemented Interfaces:
    io.spine.server.Closeable , io.spine.server.storage.Storage , java.io.Closeable , java.lang.AutoCloseable 
    @SPI() 
public class AggregateStorage<I, S extends AggregateState<I>>
extends EntityRecordStorage<I, S>

    A storage of the latest Aggregate states and the journal of the events emitted by Aggregates.

    The instances of this type solve two problems:

    • storing the latest state of each Aggregate along with its lifecycle flags and version — the source of truth for loading an Aggregate by its AggregateRepository;
    • journaling the events emitted by the Aggregates — for traceability and for the recent-history lookups such as eventHistoryBackward and the opt-in IdempotencyGuard.

    The state of an Aggregate is persisted on each update, regardless of the visibility of the Aggregate state type. This storage extendsEntityRecordStorage, so the latest states are persisted as EntityRecords and all state-related operations are served by the inherited record storage.

    End-users of the framework are able to set the visibility level for each Aggregate state by using an (entity).visibility option in the Protobuf message corresponding to the Aggregate state. For those Aggregates that are visible, the framework routines enable this storage to expose the latest states of Aggregates for further querying. To some extent, it makes this storage a part of an application's read-side. Visibility gates only this query exposure — never the state writes or the load-path reads.

    AggregateStorage supports querying the Aggregate states by the values of their declared entity columns. See io.spine.query package docs for more details on the query language.

    The events emitted by the Aggregates are appended to an intermediate EntityEventStorage. The journal is not used for restoring the Aggregate states. It narrows down the number of records to traverse when the recent history of an Aggregate is read, compared to searching the Event Store of the whole Bounded Context.

    The journal grows as the Aggregates emit events. To bound the growth, production systems run the time-based truncation as a periodic maintenance operation.

    Before the event-sourcing cutover, the journal was persisted as AggregateEventRecords — a record kind that could also hold Snapshots. The current runtime neither writes nor reads records of that kind. They remain in the underlying storage as inert data, and their proto definitions are retained so that the persisted records stay parseable.

      • readStates

         Iterator<EntityRecord> readStates(TargetFilters filters, ResponseFormat format)

        Queries the storage for the Aggregate states according to the passed filters and returns the results in the specified response format.

        The results of this call are eventually consistent with the latest states of aggregates, as instances are not restored from their events for querying. Instead, this method works on top of the storage of the latest known aggregate states, for better performance. In a distributed environment, the records in this storage may be outdated, as new events may have been emitted by an aggregate instance on other server nodes.

        Parameters:
        filters - the filters to use when querying the aggregate states
        format - the format of the response
        Returns:

        an iterator over the matching EntityRecords

      • truncate

         void truncate(Timestamp olderThan)

        Truncates the journal, deleting the events emitted before olderThan.

        An event record is deleted if and only if its event was emitted strictly before the given time. To purge the whole journal of the served Aggregates, pass the current time — or any later moment.

        Truncation bounds the recent history available to the business logic and to the opt-in IdempotencyGuard. When the guard is enabled, choose a cutoff old enough to retain the event history depth of the repository, so that the deduplication window stays intact.

        The events are removed by a query on their creation time rather than by reading the journal into memory, so the operation scales to large journals; it remains bulk maintenance rather than a per-dispatch call.

        Parameters:
        olderThan - only the events emitted strictly before this time are deleted

      • close

         void close()

        Closes this storage, releasing the inherited record storage and the event journal.
