server/io.spine.server.event/React

React

@Retention(value = RetentionPolicy.RUNTIME)
@Target(value = ElementType.METHOD)
@Documented
@AcceptsExternal
@AcceptsFilters
@AcceptsContracts
annotation class React

Marks a method of an entity as one that may modify the state of the entity in response to some domain event.

A reacting method must be annotated @React.

Like other message-handling methods, event reactors are designed to be called by the framework only. Therefore, it is recommended to declare them package-private (or internal in Kotlin). It discourages developers from calling these methods directly from anywhere. It is also acceptable to use protected if the declaring class inherits the method from a superclass.

This level of access still declares that an event reactor method is a part of the Bounded Context-level API. See the BoundedContext description on how the packages and Bounded Contexts relate.

Accepted Parameters

Each reacting method must accept an event message derived from EventMessage as the first parameter. Optionally, one may pass some additional parameters and put the incoming message into some perspective. Available sets of parameters are described below.

Single event message

@React
 EngineStopped on(CarStopped event) { ... }

An event message along with its context

The context brings some properties related to the event, such as the actor ID or the timestamp of the event emission: 

@React
 ProjectOwnerAssigned on(ProjectCreated event, EventContext context) { ... }

Rejection message with an original command message

If an event is a rejection, one may additionally specify the command message, which led to this rejection. This will act like a filter. In the example below, rejections of the CannotAllocateCargo type only caused by the rejected DeliverOrder command will be dispatched: 

@React
 OrderDeliveryFailed on(CannotAllocateCargo event, DeliverOrder command) { ... }

It is also possible to add the context of the origin command to access even more properties: 

@React
 ProjectRenameFailed on(ProjectAlreadyCompleted event, RenameProject command, CommandContext ctx) { ... }

Return Types

The essence of a reacting method is an emission of one or several events in a reaction to the dispatched event.

As long as an entity may have a complex logic of determining which event to emit in reaction, the React-marked methods allow a variety of options for the returned types. A reacting method must return one of the following.

An event message

@React
 TaskReassigned on(UserDeactivated event) { ... }

An Optional event message

@React
Optional<PersonAllowedToBuyAlcohol> on(PersonAgeChanged event) { ... }

One of particular events

Use one of the Either types to explicitly tell alternatives returned by the function. This way you can also return a special NoReaction event stating that the entity may choose not to emit an event as its reaction at all: 

@React
EitherOf3<ProjectCompleted, ProjectEstimateUpdated, NoReaction> on(TaskCompleted event) { ... }

An Iterable of event messages

@React
Iterable<StoryMovedToBacklog> on(SprintCompleted event) { ... }

A tuple of event messages

Being similar to Iterable, tuples allow declaring the exact types of returning values, including Optional values: 

@React
Pair<ProjectOwnerAssigned, ProjectDueDateChanged> on(ProjectCreated event) { ... }

@React
Pair<TaskAssigned, Optional<TaskStarted>> on(TaskAdded event) { ... }

If the annotation is applied to a method which does not satisfy either of these requirements, this method will not be registering for receiving events.
